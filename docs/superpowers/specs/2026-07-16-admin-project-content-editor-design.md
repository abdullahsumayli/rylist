# محرّر محتوى المشروع في لوحة التحكم — التصميم

**التاريخ:** 2026-07-16
**الحالة:** معتمد (بانتظار مراجعة المواصفات)
**النطاق:** لوحة تحكم الأدمن (`admin/`)

## المشكلة

لوحة التحكم تسمح بإنشاء/تعديل **بطاقة** المشروع فقط (حقول مسطّحة: كود، مدينة، سعر، صورة واحدة، بروشور، إحداثيات، عنوان/حي/وصف). أما المحتوى الداخلي الغني المخزّن في عمودَي `projects.gallery` و`projects.details` فلا يوجد له أي محرّر، فدخل عبر سكربتات seed والفريق لا يستطيع تعديله.

المحتوى بلا محرّر (كلّه معروض للعميل على صفحة المشروع):

| البيانات | العمود / المسار | الشكل |
|---|---|---|
| معرض صور المشروع | `gallery` | `[url, …]` |
| الحقائق | `details.facts` | `[{label:i18n, value:i18n}, …]` |
| الوحدات الداخلية | `details.units` | `[{title:i18n, description:i18n, price:i18n, specs:[{label:i18n, value:i18n}], gallery:[url], floorplan:url}, …]` |
| المميزات | `details.features` | `[i18n, …]` |
| نقاط الموقع | `details.location` | `[i18n, …]` |
| أنواع الوحدات | `details.unitTypes` | `[{title:i18n, detail:i18n}, …]` |

حيث `i18n = { ar, en, zh }` (البيانات الحالية تحوي `ar`/`en` فقط؛ `zh` اختياري).

مثال حجم: `NAJD-2` فيه ٤ صور معرض و٢٨ وحدة.

## الهدف / معايير النجاح

- يقدر الفريق (غير التقني، عربي أساسًا) يضيف/يعدّل/يحذف/يرتّب: صور المعرض، الوحدات وصورها ومخططاتها ومواصفاتها وأسعارها، الحقائق، المميزات، نقاط الموقع، أنواع الوحدات — بكل اللغات الثلاث.
- الحفظ لا يمسّ أي عمود أو مفتاح `details` غير الذي جرى تعديله (صفر خطر على بيانات الـseed).
- إعادة استخدام لبنات اللوحة الموجودة (تبويبات اللغة، رفع الصور) بدل تكرارها.
- بعد الحفظ، مسار النشر الحالي («نشر») يعيد بناء الصفحات دون تغيير.

## خارج النطاق (YAGNI)

- لا محرّر WYSIWYG للأسعار/الأرقام — السعر نص i18n جاهز يكتبه الفريق (مثل «٢٬٢٠٠٬٠٠٠ ريال»).
- لا تغيير على تعريف حقول البطاقة الأساسية في `entities.js`/`fields.js`.
- لا تغيير على مسار النشر أو القوالب أو الصفحة العامة.
- لا استيراد جماعي (Excel/CSV) — منفصل.

## القرار المعماري

**الخيار المعتمد (ب): محرّر محتوى مخصّص للمشروع** — منفصل عن الفورم العام.

بدائل مرفوضة:
- **(أ) توسيع الفورم العام** بأنواع حقول متداخلة — محرّر ٢٨ وحدة تطبيق مصغّر بحاله، حشره في فورم مفتاح/قيمة عام يخرّب حدود `fields.js`.
- **(ج) مربّع JSON خام** — غير صالح لمستخدم غير تقني وخطر (فاصلة واحدة تكسر المشروع).

## البنية

### وحدة جديدة: `admin/content.js`

تصدّر:

```
renderProjectContent(root, project, onDone)
```

- `project` — صف المشروع (نحتاج `id` على الأقل؛ نعيد جلب الصف كاملًا طازجًا عند الفتح).
- `onDone()` — يُستدعى بعد الحفظ أو الرجوع (يرجّع لقائمة المشاريع).

### الربط في `admin/list.js`

- إضافة زر **«المحتوى»** في صف الأزرار داخل بطاقة المشروع ([list.js:68-70](../../../admin/list.js#L68-L70)) بجانب «تعديل/حذف».
- عند الضغط: `renderProjectContent(root, row, () => renderProjects(root))`.

### إعادة الاستخدام من `admin/fields.js`

- `uploadImage(prefix, file)` — مُصدّر مسبقًا.
- `localeTabs(field, value, onLocale)` — **حاليًا داخلي؛ نُصدّره** ليُستعمل في `content.js`. (تغيير بسيط: إضافة `export`.)

لا تغيير سلوكي على `fields.js` عدا التصدير.

## نموذج البيانات والحفظ

1. **الفتح:** `sb.from('projects').select('gallery, details').eq('id', project.id).single()` — نجيب أحدث نسخة.
2. **العمل:** `draft = deepClone({ gallery: row.gallery || [], details: row.details || {} })`. كل التعديلات على `draft`.
3. **البناء:** دالة نقية `buildPayload(draft)` تُطبّع وترجّع `{ gallery, details }` — قابلة للاختبار بلا DOM.
4. **الحفظ:** `sb.from('projects').update(payload).eq('id', project.id)` — عمودان فقط. مفاتيح `details` غير الملموسة (`unitTypes`… وأي مفتاح مستقبلي) محفوظة لأننا حمّلنا `details` كاملًا وعدّلنا داخله.
5. **الأخطاء:** فشل الحفظ → `alert(error.message)` (متسق مع الكود الحالي).

## المكوّنات (لبنات مركّزة)

كل لبنة دالة ترجّع `HTMLElement` وتُعدّل مصفوفة/كائن draft عبر المرجع (in place)، تماشيًا مع نمط `fields.js`.

| اللبنة | التوقيع (مبدئي) | تعمل |
|---|---|---|
| `i18nField(value)` | value = كائن i18n | تبويبات ع/EN/中文 (تغلّف `localeTabs`) |
| `imageInput(getVal, setVal)` | — | رفع/رابط صورة واحدة (مقتطف من منطق fields.js) |
| `galleryEditor(arr)` | arr = `[url]` | صور متعددة: إضافة (رفع)، حذف، ترتيب (أعلى/أسفل) |
| `pairsEditor(arr, {aKey, bKey})` | arr = `[{label,value}]` | صفوف زوج i18n×i18n؛ إضافة/حذف/ترتيب |
| `stringsEditor(arr)` | arr = `[i18n]` | صفوف نص i18n؛ إضافة/حذف/ترتيب |
| `collapsible(title, count, bodyFn)` | — | قسم مطوي بعدّاد؛ يبني الجسم كسولًا عند الفتح |

### تخطيط الأقسام (داخل `renderProjectContent`)

```
NAJD 2 — المحتوى والوحدات            [حفظ] [رجوع]
  ▸ 📸 معرض المشروع   → galleryEditor(draft.gallery)
  ▸ 📊 الحقائق        → pairsEditor(draft.details.facts, {aKey:'label', bKey:'value'})
  ▾ 🏠 الوحدات        → unitsList(draft.details.units)
  ▸ ✨ المميزات       → stringsEditor(draft.details.features)
  ▸ 📍 الموقع         → stringsEditor(draft.details.location)
  ▸ 🏘 أنواع الوحدات  → pairsEditor(draft.details.unitTypes, {aKey:'title', bKey:'detail'})
```

عند غياب مفتاح في `details` (مشروع قديم/فاضي) نهيّئه `[]` كسولًا عند فتح قسمه، بدون كتابته إن ظل فاضيًا.

### قائمة الوحدات ومحرّر الوحدة

- `unitsList(units)` — قائمة صفوف مختصرة (عنوان الوحدة بالعربي + زر تعديل + حذف + ترتيب) و«+ وحدة».
- «تعديل»/«+ وحدة» يفتح `unitForm(unit)`:
  - `i18nField` للعنوان، الوصف، السعر.
  - `pairsEditor(unit.specs, {aKey:'label', bKey:'value'})` للمواصفات.
  - `galleryEditor(unit.gallery)` لصور الوحدة.
  - `imageInput` للمخطط (`unit.floorplan`).
  - أزرار «تم/رجوع» تعود لقائمة الوحدات (التعديلات محفوظة في draft بالمرجع).

## معالجة الأخطاء

- فشل رفع صورة: حالة inline (نفس نمط `fields.js` — `is-busy/is-ok/is-err`).
- فشل حفظ: `alert(error.message)`.
- ضمان عدم الدهس: نكتب `gallery` و`details` فقط.
- حذف عنصر: تأكيد `confirm()` قبل الحذف (متسق مع اللوحة).

## التحقّق

- **آلي:** `buildPayload(draft)` دالة نقية — اختبار وحدة يتحقق: الحفاظ على `unitTypes` والمفاتيح غير الملموسة، شكل `units`/`facts`/`features`/`location`، وعدم إدخال مفاتيح فارغة زائدة. (إن لم يوجد test runner، يُضاف مشغّل خفيف أو يُشغّل الملف عبر Node بلا DOM لأنه نقي.)
- **يدوي:** عبر لوحة التحكم على صف اختبار: فتح مشروع → إضافة وحدة + رفع صورة + مخطط → حفظ → التأكد من شكل JSON في DB → نشر → التأكد من الصفحة العامة.
