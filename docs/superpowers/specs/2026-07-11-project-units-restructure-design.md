# إعادة هيكلة صفحة المشروع + الوحدات الغنية (Project page restructure + rich units)

> التاريخ: 2026-07-11
> الحالة: مواصفات معتمدة مبدئياً — بانتظار مراجعة المستخدم قبل خطة التنفيذ.

## المشكلة (Problem)

صفحة المشروع الحالية ([templates/project.html](../../../templates/project.html) +
[scripts/lib/projectPages.mjs](../../../scripts/lib/projectPages.mjs)) تعرض المعرض في الأسفل،
و«أنواع الوحدات» كبطاقات سطر واحد (`{title, detail}`) بدون صور ولا مخططات ولا تفاصيل لكل وحدة.

المطلوب (من المستخدم، عبر صورة نجد ٢ ومربع مرسوم على طرفها):
1. **إعادة ترتيب** أقسام الصفحة لتطابق الترتيب: معرض الصور → وصف المشروع → أنواع الوحدات → مخططات الوحدات → الخرائط.
2. **الاحتفاظ** بالأقسام المفيدة الحالية غير المذكورة في المربع (تفاصيل المشروع، المزايا والمرافق، المعالم القريبة).
3. **المشروع حاوية هرمية**: جواه عدة **وحدات**، وكل وحدة كيان غني — لها صورها وشرحها وتفاصيلها **ومخططها**، مو بطاقة سطر واحد.
4. الوحدات تُعرض **داخل صفحة المشروع** (بلوكات قابلة للفتح)، لا صفحات مستقلة.
5. **بدون** قائمة تنقّل جانبية أو علوية (المربع كان لتوضيح الهيكل فقط).

## غير المطلوب (Out of scope — YAGNI)

- لا صفحات مستقلة لكل وحدة (لا قوالب/مسارات جديدة).
- لا قائمة تنقّل (sidebar/TOC).
- لا محرّر مصفوفات في لوحة التحكم للوحدات/المعرض/المخططات — تبقى بيانات الوحدات تُعبّى عبر SQL مثل المعرض والتفاصيل الحالية. (إضافة محرّر وحدات غني للوحة = ميزة مستقبلية منفصلة.)

## الهيكل النهائي لصفحة المشروع (Final section order)

بعد الهيدر (صورة البطل + العنوان + الحالة + الموقع + السعر):

| # | القسم | الحالة | يظهر متى |
|---|-------|--------|----------|
| 1 | **معرض صور المشروع** (`pgallery`) | موجود — ينتقل للأعلى | عند وجود `gallery` |
| 2 | **وصف المشروع** (`pdetail__desc`) | موجود | عند وجود وصف |
| 3 | **تفاصيل المشروع** (`pfacts`) | موجود | عند وجود `details.facts` |
| 4 | **الوحدات** (`punits` — بلوكات غنية) | جديد (يطوّر `unitTypes`) | عند وجود `details.units` أو `details.unitTypes` |
| 5 | **المزايا والمرافق** (`pfeatures`) | موجود | عند وجود `details.features` |
| 6 | **الخرائط** (`pmap`) + المعالم القريبة (`plocation`) | جديد | الخريطة تظهر دائماً؛ المعالم عند وجود `details.location` |

كل قسم يبقى **رشيقاً** (يختفي إذا فارغ) عدا الخريطة (تظهر دائماً بالحي كحد أدنى).

## نموذج البيانات (Data model)

### 4.1 أعمدة جديدة على جدول `projects` (هجرة جديدة)

```sql
alter table public.projects add column if not exists map_lat double precision;
alter table public.projects add column if not exists map_lng double precision;
```

- لا حاجة لعمود جديد للمخططات ولا للوحدات — تُخزَّن داخل `details` (jsonb موجود).
- `map_lat/map_lng` اختيارية؛ عند غيابها نرجع لبحث نصي بالحي/المدينة.

### 4.2 توسعة `details` (jsonb) — إضافة `units`

الشكل الحالي:
```jsonc
details = {
  facts:     [{ label:{ar,en}, value:{ar,en} }],
  unitTypes: [{ title:{ar,en}, detail:{ar,en} }],   // القديم — يبقى للتوافق الرجعي
  features:  [{ ar, en }],
  location:  [{ ar, en }]
}
```

الشكل الجديد (يُضاف بجانب القديم):
```jsonc
details.units = [
  {
    title:       { ar, en },                       // "تاون هاوس ٣ غرف"
    description: { ar, en },                        // شرح الوحدة
    specs:       [{ label:{ar,en}, value:{ar,en} }],// مساحة/غرف/طوابق… (اختياري)
    gallery:     [ "https://…", … ],                // صور الوحدة (اختياري)
    floorplan:   "https://…"                        // صورة المخطط (اختياري، أو floorplans:[])
  }
]
```

**توافق رجعي (Backward compatibility):** المُصيِّر (renderer) يفضّل `details.units` (البلوكات الغنية).
إذا غاب `units` ووُجد `unitTypes`، يعرض البطاقات القديمة كما هي. يعني المشاريع الحالية (NAJD 1..7)
تشتغل بدون تغيير حتى نحدّث بياناتها للشكل الغني.

## سلوك العرض (Rendering behavior)

### 5.1 بلوك الوحدة (Unit block) — HTML/CSS فقط، بلا جافاسكربت

كل وحدة داخل `<details class="punit-rich">` مع `<summary>` للعنوان (accordion أصلي).
الوحدة الأولى `<details open>`. داخل كل بلوك:

- **صور الوحدة**: شبكة صور + lightbox (نفس نمط `pgallery` مع IDs فريدة لكل وحدة، مثل `#u-{code}-{ui}-{ii}`).
- **شرح الوحدة**: فقرة نص.
- **تفاصيل الوحدة**: `specs` كشبكة صغيرة (مثل `pfacts`).
- **مخطط الوحدة**: صورة `floorplan` قابلة للتكبير (نفس lightbox). عنوان فرعي «المخطط».

بهذا يندمج «مخططات الوحدات» من المربع داخل كل وحدة (مخطط الوحدة يعيش مع الوحدة، لا قسم منفصل).

### 5.2 قسم الخرائط (Maps) — تضمين Google بدون مفتاح

- إذا وُجدت إحداثيات: `https://maps.google.com/maps?q={lat},{lng}&z=15&output=embed`
- وإلا: `https://maps.google.com/maps?q={encodeURIComponent(district + " " + city)}&z=13&output=embed`
- داخل `<iframe>` بنسبة عرض ثابتة (`.pmap`), `loading="lazy"`, بلا مفتاح API.
- تحت الخريطة: قائمة «المعالم القريبة» (`details.location`) الحالية إن وُجدت.

## تعبئة البيانات (Data population)

- الشيفرة تعمل مع البيانات الفارغة (fallback رشيق). لكن لرؤية الشكل الغني فعلياً نحتاج بيانات وحدات
  (صور + شرح + مخططات لكل وحدة).
- مصدر محتمل: بروشورات NAJD الحالية (فيها معلومات الوحدات ومخططاتها). كمرجع، نعبّي **نجد ٢** أولاً
  من بروشوره (إن أمكن استخراج الصور/المخططات)، ثم البقية تتبع نفس النمط.
- الإحداثيات: مشاريع NAJD كلها في «الرمال، الرياض» → الخريطة تشتغل تلقائياً بالحي؛ نضيف إحداثيات دقيقة
  لكل مشروع لاحقاً عند توفرها.

## الاختبار (Testing / verification)

- `npm run build` ينجح بلا أخطاء.
- مشروع موجود (مثل NAJD بترتيب `unitTypes` القديم) → يُبنى بالترتيب الجديد، وقسم الوحدات يعرض البطاقات القديمة، والخريطة تظهر بالحي.
- مشروع تجريبي فيه `details.units` غنية → يعرض بلوكات الوحدات مع الصور/الشرح/التفاصيل/المخطط.
- lightbox المعرض ومخططات الوحدات يفتح/يغلق (CSS `:target`) بلا تعارض IDs.
- الخريطة تُضمَّن وتُحمّل (lazy) في المتصفح.
- RTL سليم؛ الجوال: البلوكات والشبكات تتجاوب.

## مخاطر (Risks)

- **تعارض IDs في lightbox**: عدة معارض (مشروع + وحدات) في نفس الصفحة → نضمن بادئة فريدة لكل مجموعة.
- **بيانات فارغة**: بدون بيانات وحدات غنية، الصفحة تبدو شبه الحالية (fallback) — مقبول ومقصود.
- **iframe الخريطة**: يعتمد على تضمين Google المجاني بدون مفتاح؛ لو تغيّر سلوكه مستقبلاً نراجع البديل (صورة ثابتة).
