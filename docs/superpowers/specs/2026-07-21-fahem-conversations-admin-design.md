# قسم "المحادثات" — أرشيف محادثات فهم في لوحة الأدمن

> التصميم · 2026-07-21 · rylist
> المشكلة: محادثات مستشار "فهم" لا تُحفظ في أي مكان — تُعالَج لحظياً وتُنسى، فلا يقدر المالك يقرأ وش يسأل الناس ولا يتابع مين تحوّل لطلب.

## الهدف (Done when)
- كل محادثة جديدة مع فهم تُحفظ تلقائياً من الآن فصاعداً.
- في لوحة الأدمن قسم/أيقونة جديدة "المحادثات" تعرض كل المحادثات.
- من القائمة يقدر المالك: (١) يقرأ أي محادثة كاملة، و(٢) يميّز المحادثات اللي تحوّلت لطلب (lead) ويشوف الجوال.
- القسم محمي بالأدمن مثل بقية الأقسام.

## خارج النطاق (YAGNI)
- تعبئة المحادثات القديمة (ما تُحفظ حالياً — القسم يبدأ فاضي).
- حذف تلقائي / سياسة احتفاظ / إخفاء أرقام. (يُضاف لاحقاً لو لزم.)
- بحث/فلترة متقدمة، تصدير، تحليلات. (نبدأ بقائمة + قراءة.)
- تعديل الـ edge function `fahem-chat`.

## القرارات المعتمدة
- **آلية التخزين: من الودجت (client-side)، بنمط "إضافة فقط".** الودجت يضيف صفًّا بعد كل رد يحمل النص الكامل حتى تلك اللحظة. لا تعديل على الـ edge function.
  - **لماذا إضافة-فقط وليس صف-واحد بالـ upsert؟** في PostgreSQL أي `UPDATE ... WHERE session_id=…` يقرأ العمود، وهذه القراءة تخضع لسياسة **SELECT**. إعطاء `anon` صلاحية SELECT يعني أن أي حامل لمفتاح anon يقرأ كل المحادثات (تسريب أرقام). لذا نتجنّب UPDATE تمامًا: `anon` يملك INSERT فقط (بلا SELECT)، والأدمن يجمع الصفوف حسب `session_id` ويأخذ الأكمل. الثمن: عدة صفوف لكل محادثة (تُجمَّع في العرض) — مقبول لأداة داخلية منخفضة الحِمل.
- **أولوية العرض: قراءة المحادثات + متابعة الطلبات معاً** — قائمة تجمع الاثنين مع تمييز المحوّلة لطلب.
- **الخصوصية:** النصوص تُحفظ كاملة (تشمل الأرقام أحياناً). القسم محمي بالأدمن، فلا إخفاء الآن.

## المعمارية

### ١) قاعدة البيانات — جدول `fahem_conversations` (إضافة فقط)
| الحقل | النوع | الوصف |
|------|------|------|
| `id` | `uuid` (PK) | مفتاح الصف `gen_random_uuid()` |
| `session_id` | `uuid` | معرّف المحادثة — يولّده الودجت مرّة عند فتح صفحة فهم (ليس مفتاحًا) |
| `transcript` | `jsonb` | مصفوفة الرسائل `[{role, content, properties?}]` كاملة حتى تلك اللحظة |
| `lang` | `text` | `ar` / `en` |
| `msg_count` | `int` | عدد الرسائل (للعرض واختيار الصف الأكمل) |
| `first_message` | `text` | أول رسالة عميل (معاينة في القائمة) |
| `became_lead` | `bool` | صار طلباً؟ (يُضبط في الصف المكتوب بعد نجاح نموذج التواصل) |
| `phone` | `text` | جوال العميل إن تركه |
| `created_at` | `timestamptz` | وقت كتابة الصف (`default now()`) |

- **إضافة فقط:** كل دور محادثة يُنشئ صفًّا جديدًا يحمل النص الكامل حتى تلك اللحظة. الأدمن يجمع حسب `session_id` ويأخذ الصف الأكبر `msg_count` (الأكمل)؛ `became_lead`/`phone` تُجمَّع عبر صفوف الجلسة.
- فهرس على `(session_id, created_at desc)`.

### ٢) RLS
- **INSERT:** لدور `anon` (الودجت) بـ `with check (true)` — نفس نمط `anon insert leads` تمامًا (سابقة مقبولة في المشروع).
- **SELECT + DELETE:** للأدمن فقط عبر `is_admin()` (نمط `leads`).
- **لا anon SELECT ولا anon UPDATE** — هذا ما يحفظ الخصوصية (لا قراءة عامة للمحادثات).

### ٣) الودجت — `assets/js/fahem-chat.js`
- توليد `session_id = crypto.randomUUID()` مرّة في حالة الوحدة (module state) عند الحاجة (`ensureSession`، مع بديل لو غاب `crypto`).
- دالة `persistConversation()`: تبني الحمولة من `msgs` (النص + properties المختصرة) وتعمل `sb.from("fahem_conversations").insert(payload)` — **إضافة فقط**.
- تُستدعى بعد كل رد مساعد (نهاية `pushAssistant`) — best-effort، أي فشل يُبتلع بصمت ولا يعطّل المحادثة.
- لا تحفظ زيارة بلا أي رسالة عميل (تفادي صفوف فارغة لكل فتح للصفحة).
- عند نجاح `submitContact`: تضبط `leadCaptured=true` و`leadPhone`، ثم تضيف صفًّا جديدًا يحمل `became_lead=true` والجوال (يصير هو الأكمل، فيظهر في العرض كمحوّل لطلب).
- `created_at` يُدار من الـ DB.

### ٤) واجهة الأدمن — `admin/conversations.js` (جديد) + تسجيل في `app.js` و`shell.js`
- **أيقونة جديدة** `chat` في خريطة `ICONS` بـ [shell.js](admin/shell.js) (فقاعة حوار: `<path d="M21 15a2 2 0 0 1-2 2H7l-4 4V5a2 2 0 0 1 2-2h14a2 2 0 0 1 2 2z"/>` بنمط مختلف عن `contact`، مثلاً مع نقاط داخلية).
- **تسجيل القسم** في `buildSections()` بـ [app.js](admin/app.js):
  `{ key: "conversations", label: "المحادثات", icon: "chat", render: (v) => renderConversations(v) }`
  (يوضع بعد "الطلبات" مباشرة).
- **`renderConversations(root)`** (نمط [leads.js](admin/leads.js)):
  - يقرأ `sb.from("fahem_conversations").select("*").order("created_at", { ascending: false })` ثم **يجمع حسب `session_id`** (يأخذ أكبر `msg_count`؛ `became_lead`/`phone` تجميع OR/coalesce).
  - **حالة فاضية** أنيقة: "لا توجد محادثات بعد — ستظهر هنا فور بدء أول عميل محادثة مع فاهم."
  - **جدول:** التاريخ (`last_at`) · اللغة · عدد الرسائل · أول سؤال (`first_message` مقصوص) · شارة "↩ طلب" + الجوال لو `became_lead`.
  - **الصف قابل للفتح:** ضغط الصف يوسّع صفاً تحته يعرض `transcript` كفقاعات (عميل يمين / فهم يسار، بنمط بسيط قابل لإعادة الاستخدام من CSS الأدمن)، وبطاقات العقارات كأسطر مختصرة (كود + عنوان + سعر).

## تدفّق البيانات
```
عميل يكتب في fahem.html
  → الودجت يرسل التاريخ للـ edge (كما هو، بلا تغيير) → يرد فهم
  → الودجت: pushAssistant() → render() → persistConversation() → upsert في fahem_conversations
  → (لو ترك جواله) submitContact() → insert في leads + update المحادثة (became_lead, phone)

الأدمن يفتح قسم "المحادثات"
  → renderConversations() يقرأ fahem_conversations (RLS: أدمن فقط)
  → قائمة → فتح صف → عرض transcript كامل
```

## معالجة الأخطاء
- الحفظ best-effort: فشل الشبكة/الـ RLS لا يعطّل تجربة العميل (try/catch صامت، مثل نداءات الودجت الحالية).
- الأدمن: خطأ القراءة يظهر كـ `admin-err` (نمط موجود).
- `transcript` تالف/فارغ في العرض: نتعامل معه بأمان (مصفوفة فاضية → لا فقاعات).

## الاختبار (يدوي — الموقع static بلا إطار اختبار)
1. افتح `fahem.html`، سوِّ محادثة (سؤال + بحث + عرض عقار) → تأكد من ظهور صف في الجدول عبر SQL/الأدمن.
2. اترك جوالاً في نموذج التواصل → تأكد `became_lead=true` و`phone` مضبوط ونفس الجوال ظهر في "الطلبات".
3. افتح قسم "المحادثات" في الأدمن كأدمن → القائمة صحيحة، وفتح الصف يعرض الحوار كامل.
4. افتح كـ non-admin (أو بلا تسجيل) → RLS يمنع القراءة (لا تسريب).
5. جرّب محادثتين متوازيتين (تبويبين) → session_id مختلف → صفّان منفصلان.
